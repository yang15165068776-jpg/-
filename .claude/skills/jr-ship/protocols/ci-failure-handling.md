# CI-failure handling — `/jr-ship`

**Canonical source** for `/jr-ship`'s CI-failure investigate-and-fix procedure. `jr-ship/SKILL.md` reads this file into lead context at Phase 1 (under the hard-fail + non-empty + smoke-parse guard, alongside the `shared/*.md` files) and applies it at Phase 3a step 13 and Phase 3b step 11a-multi. Update here to update the procedure.

Invoked when the CI-watch (`gh pr checks <number> --watch --fail-fast` on GitHub; `glab ci status --live -b <branch>` on GitLab — a semantic adaptation per `../../shared/forge-detection.md`) reports a **failed check/job**. (The 10-minute **timeout** path is separate and unchanged — report status and stop.) The lead stays mechanical; the diagnosis-and-fix work is delegated to an `opus` sub-agent. Run at most **2 fix cycles**, then stop.

One cycle:

1. **Spawn the CI-fix agent** — use the `Agent` tool with `model: "opus"` (or the `--model` override when set — `../../shared/model-override.md`) and no `subagent_type` (default general-purpose, same as the split-analysis agent — no plugin dependency). The prompt includes: the detected forge (`FORGE`/`CLI` per `../../shared/forge-detection.md`), the PR/MR number, the branch name, and the CI-watch failure summary; the instruction "diagnose the root cause exhaustively before editing — do not patch symptoms; if the failure is not a code-level issue (flaky test, infra, permissions), say so instead of editing; when the diagnosis rests on an external fact (a tool's behavior, a version requirement, an API change), confirm it against an authoritative source per `../../shared/claim-verification.md` (raw `.md` / `gh api`, never a lone WebFetch summary) before proposing a fix — do not assert version or behavior facts from memory"; the full `../../shared/untrusted-input-defense.md` block verbatim (loaded in Phase 1 — CI logs are untrusted external input); and the full `../../shared/code-edit-discipline.md` block (also loaded in Phase 1) prefixed with the CI-fix-specific lead-in "Your assignment is to fix the CI failure described above. The discipline below applies to every line you change:" and with the canonical's title and opening sentence elided (skip from `# Code-edit discipline` through `…or specific simplifications to make.` only — the title and that first sentence assume a findings-list workflow that doesn't fit CI-failure repair; KEEP the next sentence "Your changes must be **surgical**: every line you modify should trace directly to your assignment." as the framing line that ties the Do/Don't list together — the lead-in just redefined "your assignment" as "the CI failure", so the framing applies cleanly — then continue through `## Do NOT` to end of file). The agent has full tools — it fetches its own CI logs per the detected forge (GitHub: `gh run list --branch <branch> --json databaseId,name,conclusion,headSha`, then `gh run view <id> --log-failed`; GitLab: `glab ci status -b <branch>` then `glab ci trace`/`glab ci get` for the failed job — a semantic adaptation, not a rename, per `../../shared/forge-detection.md`), reads the repo, and edits files in the working tree in place. It reports back its diagnosis plus the change it made, **or** "no code-level fix — cause is <X>". **Headless mode** (per `../../shared/secret-scan-protocols.md` "Headless/CI detection" — no user available for the step-3 gate): instruct the agent to **diagnose only — do not edit files**.
2. **Re-scan for secrets** — run the Phase 1 step 4 secret scan (canonical patterns from `../../shared/secret-patterns.md`, already loaded) against the agent's edits (`git diff`). If a strict-tier pattern matches, halt per the Phase 1 step 4 halt protocol — do NOT commit or push, and treat it as a **non-apply outcome**: the lead reverts the agent's edits per the Step 3 clean-tree guarantee so the detected secret never lingers in the working tree. (The agent's diagnosis stays in the transcript as the record of what was attempted.)
3. **Confirm gate** — print the diagnosis and the agent's `git diff`, then AskUserQuestion: `CI failed on PR #<n>: <one-line diagnosis>. Proposed fix shown above. Options: [Apply fix + re-push] / [Skip — report and stop]`. If the agent reported "no code-level fix", skip the gate (nothing to apply) and go straight to reporting the diagnosis.
   - **Clean-tree guarantee**: on every non-apply exit — **Skip**, "no code-level fix", headless diagnose-only, or a Step 2 secret-halt — the lead MUST revert any uncommitted agent edits before returning: `git status --porcelain` to list touched files, then `git checkout -- <files>`. `/jr-ship` must never exit leaving the agent's unapproved edits dirty on the branch. (Only the **Apply** path keeps the edits — by committing them.)
4. **Apply** (on **Apply fix + re-push** only) — commit the edits with a concise `fix: <summary>` conventional-commit message (omit any `Co-Authored-By: Claude` trailer — same rule and rationale as the step 8 and step 7-multi commits). Push the branch to `origin`.
5. **Re-watch** — re-run the CI-watch (`gh pr checks <number> --watch --fail-fast` on GitHub; `glab ci status --live -b <branch>` on GitLab). The re-pushed pipeline/workflow run takes a few seconds to register, so a brief window where the watch still shows only the *old* failed checks/jobs is NOT "still failing" — wait for the new run to appear and complete. If CI is now **green**, return success to the caller. If it **fails again**, start the next cycle from step 1 (max 2 fix cycles total).

   **2-cycle cap — single-fire stuck-loop advisor**: when the 2nd fix cycle also ends non-green, before reporting failure, call `advisor()` once (no parameters — the full transcript is auto-forwarded). Per `../../shared/advisor-criteria.md`, a retry loop that exhausts its max-retry count is the canonical stuck signal — the advisor sees both fix attempts and the CI logs that kept failing and can spot a non-obvious cause. Guard the call with a `ciFixAdvisorFired` boolean set `true` after it fires, so a later invocation in the same `/jr-ship` run (e.g. another sub-PR in Phase 3b) does NOT re-fire it. If the advisor concurs (no actionable insight), report the latest diagnosis + the failure URL and stop. If the advisor offers a concrete actionable insight: in **interactive mode**, surface it via AskUserQuestion `Advisor flagged a CI-fix concern: <one-line summary>. Options: [Apply suggested fix + re-push — one more cycle] / [Stop — report and stop]`; on **Apply**, run one additional cycle (steps 1–5) with the advisor's suggestion as targeted instruction to the CI-fix agent, and if that cycle also fails do NOT re-fire — report and stop. In **headless mode** (no user for the gate), log the advisor's insight alongside the diagnosis in the report and stop.

**Outcome returned to the caller:**

- **Green** — CI passed within the fix cycles. The caller continues normally (Phase 3a → step 14/15; Phase 3b → next PR in the wait pass).
- **Not green** — fix skipped, no code-level fix, secret-halt, or the 2-cycle cap was reached (advisor consulted). The caller treats this exactly like the pre-existing CI-failure stop: Phase 3a skips cleanup (step 15); Phase 3b reports and does not enter 11b-multi.
